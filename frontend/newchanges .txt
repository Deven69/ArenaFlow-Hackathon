THE PROBLEM WITH YOUR CURRENT STACK
Supabase, Vercel, and Twilio are all non-Google. The evaluators will notice. The instruction says "effective use of Google Services" — if your database, hosting, and notifications are all non-Google, you lose points even if your Gemini usage is excellent.
The good news: you do not need to replace Supabase entirely. You need to add Google services on top of what exists, not rebuild from scratch.

THE ADDITIONS — PRIORITISED BY IMPACT ON SCORE

PRIORITY 1 — Replace Vercel with Firebase Hosting (30 minutes, zero code change)
This is the easiest swap and immediately adds a Google service to your deployment. Firebase Hosting is free, supports Next.js static export, and gives you a web.app domain which signals Google ecosystem fluency to evaluators.
Change needed: Add firebase.json and .firebaserc to your repo. Run firebase init hosting in Antigravity. Change your deploy command from vercel to firebase deploy. Your Next.js app needs output: 'export' in next.config.js for static hosting, OR use Firebase App Hosting (preview feature) which supports full Next.js SSR.
Score impact: High. Every evaluator will see the deployment URL.

PRIORITY 2 — Replace Twilio WhatsApp with Firebase Cloud Messaging + Google Cloud Functions (2 hours)
Currently your gate nudge goes via Twilio WhatsApp. Swap the delivery mechanism to FCM (Firebase Cloud Messaging) for in-app push notifications, and keep WhatsApp as a fallback only. The nudge generation still uses Gemini — you keep that Google service. But now the delivery is also Google.
More importantly: move the nudge-sending Edge Function from Supabase to Google Cloud Functions (free tier: 2 million invocations/month). The geofence cron that checks fan proximity and fires notifications becomes a Google Cloud Scheduler job triggering a Google Cloud Function. This replaces the Supabase Edge Function cron with a fully Google-native equivalent.
Change needed: Add Firebase Admin SDK to your project. Create a Cloud Function nudge-dispatcher that reads gate data from Supabase, calls Gemini for message generation, and sends via FCM to fans whose geofence triggered. Set up Cloud Scheduler to trigger it every 60 seconds during event hours.
Score impact: Very high. Cloud Functions + Cloud Scheduler + FCM is three Google services in one feature.

PRIORITY 3 — Add Google Maps JavaScript API for Gate Navigation (3 hours)
Currently your navigate-to-seat feature is text-based directions. Replace it with an actual Google Maps embed showing the walking route from the fan's GPS location to the recommended gate. This is one of the most visually impressive additions you can make and it directly solves the "crowd movement" problem from your problem statement.
Implementation: On the fan's ticket back face, the "Navigate to Seat" button opens a full-screen Google Maps view using the Maps JavaScript API. The map shows the venue, the fan's current location (browser geolocation API), and a walking route polyline to the recommended gate (the one with the shortest queue, determined by the admin dashboard data). The route automatically updates if the recommended gate changes due to a new crowd surge.
This also powers the geofence detection properly — instead of a manual proximity check in a cron job, use the Google Maps Distance Matrix API to calculate exact walking time from fan's location to each gate. This becomes the data source for the "which gate is fastest" logic that Gemini uses to generate the nudge message.
Change needed: Add @googlemaps/js-api-loader package. Create a GateNavigator component. Add Maps API key to environment variables. Add one API route /api/navigation/gate-times that calls Distance Matrix API with fan coords and all gate coords, returns walking minutes per gate.
Score impact: High. Directly visible in demo. Directly solves the stated problem.

PRIORITY 4 — Add Google Cloud Vision API as Fallback for Ticket Scan (1 hour)
You already use Gemini Vision for ticket scanning. Add Google Cloud Vision API as the preprocessing step before Gemini. Cloud Vision handles the barcode/QR detection and extracts the raw barcode number first. Then Gemini processes the full image for structured data extraction. This creates a two-step pipeline that is more reliable AND uses two distinct Google AI services.
The practical benefit: Cloud Vision's barcode detection is more reliable than asking Gemini to find and read a barcode in an image. Gemini is better at understanding the surrounding context (match name, venue, section). Use each for what it does best.
Change needed: Add @google-cloud/vision package. In /api/tickets/scan, call Cloud Vision first to extract the barcode string, then pass both the barcode value AND the image to Gemini with "Here is the barcode number already extracted: [X]. Now extract the rest of the ticket details from this image."
Score impact: Medium-high. Shows sophisticated multi-service AI architecture.

PRIORITY 5 — Add Google Analytics 4 to the PWA (20 minutes)
Dead simple. Add GA4 tracking to every screen transition. Track: ticket_scanned event, group_created event, group_joined event, checkin_completed event, nudge_received event, food_ordered event. This is one line of setup and five event calls throughout the app.
Why it matters for the hackathon: the README can mention "operational analytics via Google Analytics 4" and evaluators can verify it in the source code. It shows you think about product metrics, not just features.
Change needed: npm install @next/third-parties. Add <GoogleAnalytics gaId="G-XXXXXXXXXX" /> to your root layout. Add five gtag('event', ...) calls at key user actions.
Score impact: Low effort, positive signal on code quality and real-world thinking.

PRIORITY 6 — Add Google Translate API for Multi-Language Support (2 hours)
Your current nudge is Hinglish (Hindi-English). Add Google Translate API to support Marathi as well — Mumbai's primary local language. The fan sets their language preference during onboarding (English / Hindi / Marathi). The Gemini nudge is generated in English first, then translated via Google Translate API if the fan's preference is Hindi or Marathi.
This directly addresses the "Accessibility — inclusive and usable design" evaluation criterion. Mumbai has a large Marathi-speaking population that does not use English or Hindi as primary language. This addition scores on both Google Services AND accessibility simultaneously.
Change needed: Add one API route /api/translate that calls Google Translate API. In the nudge sending logic, check fan's language preference and translate before sending. Add language preference selector on onboarding screen (three pill buttons: EN / HI / MR).
Score impact: Medium. Scores on two evaluation criteria simultaneously.
